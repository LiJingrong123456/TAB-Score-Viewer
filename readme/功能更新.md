## (2026-07-24) - 调音器 + 预备拍 (v2.7.0)

新增独立调音器 + GTP 模式预备拍倒计时。

### 调音器

**核心模块** `tuner.py` (~600 行): `StringSpec` / `GUITAR_STRINGS` (6 弦 A4=440Hz 12 平均律) / `detect_pitch` (自相关法 + 抛物线插值, ±1 cent) / `TunerWorker(QThread)` (sounddevice 44100Hz, 20 fps) / `TunerDial` (圆形 -50~+50 cent 表盘, 绿色 in-tune 扇区) / `StringIndicator` (单弦灯色 + cent) / `TunerDialog` (主窗口)。

**算法精度** (纯正弦波 1s 缓冲): 6 弦偏差 < 5 cent; 220Hz 不混淆到 110Hz; 静音/短缓冲/None 安全返回 0。

**三入口**: `DisplayWindow` 工具栏 `ModernButton` / `SelectionWindow` 文件夹栏 `ModernButton` / `Ctrl+T` (新增第 12 个 `DEFAULT_SHORTCUTS`)。**跨类单例**: 模块级 `_tuner_dialog_singleton`, 避免多窗口抢麦克风。

### 预备拍倒计时

**需求**: GTP 歌曲流全新开始前插入 3 秒停顿, 让用户有时间准备 (调音/拿琴/看谱)。

**核心组件**:
- `class CountdownOverlay(QWidget)` - 浮在 `DisplayWidget` 之上的半透明覆盖层, 中央大数字 (3/2/1 白色 → GO 绿色), 顶部提示, 底部取消说明
- `_start_countdown()` / `_on_countdown_tick()` / `_finish_countdown()` / `_after_go_signal()` / `_cancel_countdown()` 状态机
- `_do_actual_playback()` - 抽出原 `start_playback` 核心逻辑, 倒计时结束共用

**触发条件** (仅 GTP 全新开始):
- ✅ GTP 模式 + `gtp_player.is_audio_ready` + 非暂停恢复
- ❌ 暂停恢复 (`is_paused=True`) - 走 `resume()` 立即继续
- ❌ 图片/PDF 模式 - 无需预备拍
- ❌ 节拍器播放 - 立即开始

**节拍器协同** (v2.7.0 改进):
- **倒计时开始前**: `gtp_player.stop()` 完全重置, 停止所有音频 (含正在独立播放的节拍器)
- **倒计时结束**: 显式调 `set_metronome(enabled, vol, gain)` 同步配置, 然后 `play()` 内部按状态自动启停
  - `_metronome_enabled=True` → 歌曲流 + 节拍器同步启动
  - `_metronome_enabled=False` → 仅歌曲流
- **位置重置**: `_paused_time_ms = 0.0`, 全新开始从歌曲头播放

**倒计时时序**:
1. t=0s: `gtp_player.stop()` 停所有音频 → 启动 `_countdown_timer` (1000ms), 显示 "3"
2. t=1s: 显示 "2"
3. t=2s: 显示 "1"
4. t=3s: 显示 "GO" (绿色) + 停止 1s 定时器
5. t=3.5s: 隐藏覆盖层 + `set_metronome()` 同步 + `play()` 真正开始 (歌曲 + 节拍器)

**取消机制** (双保险):
- 倒计时期间再次点击播放按钮 → `_cancel_countdown()` (主入口 `start_playback` 检查)
- 倒计时期间按 `Esc` 键 → `_cancel_countdown()` (`keyPressEvent._legacy_keypress` 检查)
- 窗口关闭 (`closeEvent`) → 自动停止

**降级处理**: sounddevice/numpy 缺失 → `lazy import` + `QMessageBox`; 麦克风权限拒绝 (macOS) → `error` 信号 + 红字状态栏; RMS < -46 dBFS 视为静音。

## (2026-07-23) - 修复 Mac 平台 Cmd/Ctrl 显示错位 + 单独修饰键误报 (v2.5.2)

### 三次修复合并记录

#### 第一次 (v2.5.2 first pass) - 修饰键单独按下时弹"max 3 keys"误报

**问题**: Mac 上按 Control 键，快捷键捕获对话框错误提示"最多由 3 个按键组成"；按住修饰键不松开时错误对话框反复弹出。

**根因**: Mac Qt 平台 bug — `event.modifiers()` 按修饰键时报告**所有 4 个标志位**（系统全局状态），导致 `event_to_sequence` 输出 `Ctrl+Shift+Alt+Cmd` 触发键数超限。autorepeat 事件未过滤。

**修复**:
- `shortcuts.py` 新增 `_MODIFIER_KEY_MAP`（`Qt.Key_Control/Shift/Alt/Meta → 显示名`），`event_to_sequence` 在 `event.key()` 是修饰键时**只输出该修饰键**，忽略 modifiers 中的其他标志位。
- `shortcuts_dialog.py::keyPressEvent` 增加 `event.isAutoRepeat()` 过滤。

#### 第二次 (v2.5.2 second pass) - Mac Qt 互换 Ctrl/Cmd 语义

**问题**: Mac 按物理 Control 键（^），对话框却显示 "Cmd"；按物理 Command 键（⌘），对话框显示 "Ctrl"。

**根因**: Qt 在 macOS 上互换 Ctrl 和 Cmd 语义（`Qt.Key_Control` = 物理 ⌘，`Qt.Key_Meta` = 物理 ^）。需要在代码里反向还原为"用户物理按键名"。

**修复**:
- `_build_modifier_key_map` Mac 分支 swap：`Key_Control → "Cmd"`, `Key_Meta → "Ctrl"`。
- `event_to_sequence` Mac 分支 mod_pairs 同样 swap。
- `_MODIFIER_SORT_KEY` 跨平台保持 `Ctrl(0) → Shift(1) → Alt(2) → Cmd/Meta/Win/Super(3) → CapsLock(4) → Tab(5)`。
- `_IS_MAC` 常量上移到 `_MODIFIER_SORT_KEY` 之前。

#### 第三次 (v2.5.2 third pass) - 单独按修饰键不弹错误，等待非修饰键

**问题**: 前两次修复后,Mac 按任意修饰键都立即弹"最少有一个非修饰键"错误, QMessageBox 抢走焦点并退出 capture 模式,用户根本没机会按非修饰键。

**根因**:
1. `keyPressEvent` 在 modifier-only 事件时立即 `_show_invalid("modifier only")` + `event.accept()` 返回, QMessageBox 弹窗导致编辑器失焦, `focusOutEvent` 退出 capture 模式。
2. Mac modifier press 事件 `event.modifiers()` 报告所有 4 个 flag (bug), `event_to_sequence` 输出 `"Ctrl+Shift+Alt+Cmd"` (4 修饰键),即便后续按非修饰键,合成的序列也是 5+ 部分触发"max 3 keys"。
3. Mac 上按多个 modifier 时, `event_to_sequence` 用 early return 只返回刚按的那一个 (如按 Shift 只返回 "Shift"),如果替换 `_held_modifiers` 会丢之前的 Cmd。

**修复**:
1. **引入 `_held_modifiers: Set[str]` 状态** (在 `ShortcutCaptureEditor`):
   - 按 modifier-only 事件时: **追加** (不是替换) 到 `_held_modifiers`, 不弹错误, 保持 capture 模式
   - 按非修饰键时: 用 `_held_modifiers` (不是 event.modifiers()) + 主键合成最终序列, 校验
   - **完全绕过 Mac modifier press bug** (不依赖 `event.modifiers()` 的 flag)
2. **新增 `keyReleaseEvent`**: 释放 modifier 时从 `_held_modifiers` 移除, 支持"按 Cmd → 释放 Cmd → 按 Z" 得到 "Z" 而非 "Cmd+Z"。
3. **新增 `_show_held_progress()`**: 在 capture 模式下显示进度, 如 `"Cmd + Shift + ..."`, 用户清楚当前已按了哪些 modifier。
4. **`_enter_capture_mode` / `focusOutEvent` / `cancel_capture` / Esc / Backspace / Enter 路径**: 都清理 `_held_modifiers` 防止状态泄漏。

### 验收（v2.5.2 合并）

13 项验收场景全部通过（含 Mac 按 ⌘/^/⌥/⇧ 静默进入 `_held`、跨平台排序、autorepeat 过滤等）。详见 git log 与 `/tmp/test_v252_fourth_pass.py`。

### 修改文件

- `shortcuts.py`：新增 `_IS_MAC` / `_MODIFIER_KEY_MAP`（含左右键）/ Mac 分支 swap；`event_to_sequence` 四次修正（early return + Qt.Key_unknown + KeyboardModifiers int 转换 + 入口显式识别）。
- `shortcuts_dialog.py`：`keyPressEvent` 用 `_held_modifiers` 状态机 + 不依赖 `event.modifiers()` bug 数据；新增 `keyReleaseEvent` + `_show_held_progress`；多条清理路径。

#### 第四次 (v2.5.2 fourth pass) - 入口处显式识别修饰键，彻底绕开 Mac bug

**问题**: 前三次修复后, 在某些 Mac 平台/键盘布局下, 单按任意修饰键 (Cmd/Ctrl/Option/Shift) 仍会立即弹出"必须包含至少 1 个非修饰键"错误, 用户根本没机会按非修饰键。

**根因**: 之前的修复依赖 `event_to_sequence` 在内部判断 modifier press 事件, 但:
1. `event_to_sequence` 内部的 Mac bug 修正依赖 `event.key() in _MODIFIER_KEY_MAP` 这一前提, 若 PyQt5 版本无 `Key_Control_L/R` / `Key_Meta_L/R` 等左右键常量, 修饰键可能不被 `_MODIFIER_KEY_MAP` 覆盖
2. 修正 #3 (key=0/Key_unknown 分支) 依赖 `event.modifiers() & all_mod_flags` 判断, 但某些 Mac 配置下 modifier flags 可能为 0
3. `keyPressEvent` 中按 `event_to_sequence` 输出再判断 `if not non_mods` 有先后依赖, 若 `event_to_sequence` 返回了带主键的字符串就会进入"非修饰键"路径

**修复**: **把修饰键识别前移到 `keyPressEvent` 入口, 不再依赖 `event_to_sequence`**:
1. **优先级最高**: 若 `event.key() in _MODIFIER_KEY_MAP` → 一定按修饰键处理, 直接 `add` 到 `_held_modifiers`, 不调用 `event_to_sequence`
2. **兼容 key=0/Key_unknown**: 若 `event.key()` 是 0 或 `Key_unknown`, 但 `event.modifiers() & all_mod_flags != 0` (Mac bug 下经常是 all 4) → 启发式映射 (ControlModifier→Cmd, MetaModifier→Ctrl, AltModifier→Alt, ShiftModifier→Shift)
3. 移除 `if not non_mods:` 的死代码 (该分支永远不可达, 之前是开发遗漏)
4. 错误诊断简化: `parse_sequence` 失败只可能是 `max 3 keys`, 不需要再细分

**效果**: 任何 Mac 平台/键盘布局下, 单按修饰键都静默进入 `_held_modifiers`, 不会触发任何错误; 后续按非修饰键才合成最终序列。
